# 1. Purpose

The Event Dictionary defines every behavioral event generated by the simulator.

It serves as the official tracking specification for developers, analysts, and data engineers.

Every event documented here includes:

- business meaning
- trigger condition
- lifecycle state
- required properties
- optional properties
- generation constraints
- downstream analytical usage

This document acts as the single source of truth for all event tracking.

---

# 2. Event Tracking Philosophy

The simulator follows an event-driven architecture.

Each event represents one completed business action.

An event must satisfy the following principles:

- Atomic (one business action per event)
- Immutable (cannot be modified after generation)
- Timestamped
- Traceable to a single user
- Traceable to one session when applicable
- Valid within the user's lifecycle state

Events are generated only by simulation engines and never manually inserted.

# 8. Event Naming Convention

Event names follow:

verb_object

Examples

signup

login

dashboard_view

feature_used

trial_started

payment_success

Rules

- lowercase only
- snake_case
- past tense avoided
- no abbreviations

---

# 9. Event Validation Rules

Every event must satisfy:

✓ valid user_id

✓ valid timestamp

✓ allowed lifecycle state

✓ valid event ordering

✓ unique event_id

✓ valid session reference (if applicable)

---

# 10. Dirty Data Injection Rules

Dirty data may introduce:

- duplicate events
- missing events
- timestamp drift
- invalid casing
- orphan session reference
- delayed tracking
- impossible event ordering

These anomalies are injected only after clean simulation completes.

# 13. Event Versioning Policy

All event names are considered stable APIs.

Rules:

- Existing event names must never be renamed.
- Deprecated events remain documented until the next major version.
- New events require updates to this Event Dictionary before implementation.
- Breaking tracking changes require a major version increment.
