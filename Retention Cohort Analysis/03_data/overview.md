# 1. Purpose

This document provides a high-level overview of every dataset generated by the Synthetic SaaS User Behavior Simulator.

Rather than describing individual columns, this document explains:

- what each dataset represents,
- how datasets relate to one another,
- how they are generated,
- and how they should be used during analytics.

- event_dictionary.md

# 2. Data Philosophy

The simulator follows a layered data architecture similar to modern SaaS analytics platforms.

Instead of generating only one event table, the simulator produces multiple datasets that represent different stages of the analytical workflow.

The generated datasets are divided into three logical layers:

• Operational Data

Raw behavioral records generated directly by the simulation.

• Analytical Data

Derived datasets used for SQL analysis and BI dashboards.

• Metadata

Simulation metadata, validation reports, and execution logs.

This separation mirrors common data warehouse practices.

# 7. Data Lineage

Data lineage describes how each dataset is derived throughout the simulation and analytics pipeline.

Unlike the execution pipeline, data lineage focuses on dataset dependencies rather than software modules.

The lineage ensures that every analytical table can be traced back to its original operational data.

---

## Analytical Dependency

Most Product Analytics metrics originate from the event log.

```text
events.csv

│

├── Retention

├── Churn

├── Funnel

├── Aha Moment

├── Feature Adoption

├── DAU / WAU / MAU

├── Session Analytics

└── User Journey
```

The event log serves as the primary behavioral dataset and should be considered the single source of truth for user interaction analysis.

---

## Revenue Dependency

Revenue metrics are generated independently from behavioral analytics but remain linked through subscription data.

```text
subscriptions.csv

↓

revenue.csv

↓

MRR

ARR

ARPU

Expansion Revenue

Contraction Revenue

Revenue Dashboard
```

This separation allows financial metrics to evolve independently while maintaining consistency with subscription events.

---

## SQL Transformation Lineage

Operational datasets are never queried directly in Tableau.

Instead, they are transformed into analytics-ready tables.

```text
events.csv
      │
      ▼
SQL Cleaning
      │
      ▼
SQL Transformation
      │
      ├──────────────┐
      ▼              ▼
Retention      Aha Moment
      │              │
      ▼              ▼
Churn        Funnel
      │              │
      └──────┬───────┘
             ▼
       Tableau Workbook
```

This approach separates raw operational data from business-ready analytical datasets.

---

## Lineage Principles

The simulator follows the following lineage principles:

- Every analytical dataset must originate from operational datasets.
- Operational datasets are never manually edited.
- Derived tables are regenerated through SQL transformations.
- Tableau consumes only analytics-ready tables.
- Metadata and QA reports are generated independently and do not alter analytical datasets.
