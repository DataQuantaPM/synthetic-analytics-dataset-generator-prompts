# 1. Purpose

The Generator Layer is responsible for transforming business rules into synthetic datasets.

Each generator creates exactly one primary business entity while consuming datasets generated by upstream generators.

The Generator Layer is intentionally modular so that individual generators can evolve independently without affecting the overall simulation architecture.

---

# 2. Generator Philosophy

The Generator Layer follows six architectural principles.

## Single Responsibility

Each generator owns one business domain.

Examples

- User Generator → Users
- Session Generator → Sessions
- Event Generator → Events

---

## Deterministic

Generators produce statistically equivalent outputs when executed with the same configuration and random seed.

---

## Config-Driven

Business behavior originates from the Resolved Configuration.

Generators never hardcode business assumptions.

---

## Pipeline-Oriented

Generators execute in a predefined sequence.

Each generator consumes only published datasets.

---

## Stateless

Generators should avoid persistent mutable state.

Runtime information is provided through the Runtime Context.

---

## Observable

Each generator records metrics, execution time, warnings, and output statistics.

Each generator produces exactly one primary dataset.

```text
    │
    ▼
Internal Validation
    │
    ▼

    │
    ▼

    │
    ▼

```

Execution order is controlled by the Simulation Engine.

# 6. Generator Contracts

Each generator conforms to the Abstract Generator Contract.

Every generator must:

- receive Runtime Context
- use assigned Random Stream
- read only required datasets
- generate one primary dataset
- validate generated output
- publish dataset
- return Generator Result

Generators must never:

- invoke another generator directly
- modify upstream datasets
- modify Resolved Configuration
- bypass Dataset Registry

---

# 8. Runtime Responsibilities

Each generator is responsible for:

- consuming required datasets
- generating deterministic outputs
- collecting execution metrics
- validating generated data
- publishing datasets
- logging execution events

The following responsibilities belong outside generators:

- configuration resolution
- orchestration
- dirty data injection
- global validation
- exporting outputs

---

# 9. Generator Design Principles

## One Generator → One Dataset

Every generator owns a single primary output.

---

## Read Before Write

Generators may read published datasets but never modify them.

---

## Independent Random Streams

Each generator receives its own random stream.

Random streams are never shared.

---

## Immutable Published Data

Published datasets are treated as read-only.

---

## Contract First

Every generator follows the same runtime contract regardless of business logic.

---

## Metrics First

Every execution records:

- execution time
- rows generated
- warnings
- errors
- resource usage

---

# 10. Generator Extension Strategy

New generators should be added by extending the Abstract Generator Contract.

Example future generators include:

- Feature Usage Generator
- Notification Generator
- Support Ticket Generator
- Billing Generator
- Marketing Attribution Generator
- Experiment Generator

Extensions should integrate through the Generator Registry and Dataset Registry without requiring changes to existing generators.
