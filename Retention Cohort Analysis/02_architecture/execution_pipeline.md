# 1. Purpose

This document defines the execution order of every module in the simulator.

The execution pipeline guarantees deterministic, reproducible, and logically consistent dataset generation.

Each stage depends only on outputs produced by previous stages.

---

# 2. Execution Philosophy

The simulator follows a staged execution model.

Each stage has a single responsibility and produces immutable outputs consumed by subsequent stages.

The pipeline never skips stages.

A stage may validate its inputs before execution but must not modify outputs from previous stages.

# 6. Input → Output Contract

Every engine within the execution pipeline follows a strict input–output contract.

Each engine receives immutable inputs, performs one well-defined responsibility, and produces structured outputs for downstream engines.

No engine may modify the output generated by previous stages.

---

| Step | Engine | Input | Output | Responsibility |
|------|--------|-------|--------|----------------|
| 1 | ConfigurationLoader | YAML Configuration Files | SimulationConfig | Load and validate all simulation settings |
| 2 | UserGenerator | SimulationConfig | Users | Generate users and static attributes |
| 3 | PersonaEngine | Users | Users + Persona | Assign one persona to each user |
| 4 | HiddenVariableEngine | Users + Persona | Users + Hidden Variables | Initialize behavioral variables |
| 5 | ScenarioEngine | Users + Hidden Variables | Scenario-adjusted Users | Apply business scenarios and probability modifiers |
| 6 | LifecycleEngine | Scenario-adjusted Users | Lifecycle Timeline | Simulate weekly lifecycle progression |
| 7 | SessionEngine | Lifecycle Timeline | Sessions | Generate realistic user sessions |
| 8 | EventEngine | Sessions + Lifecycle Timeline | Events | Generate product events allowed by the current lifecycle state |
| 9 | SubscriptionEngine | Lifecycle Timeline + Hidden Variables | Subscription Timeline | Simulate trial, upgrades, renewals, downgrades, and cancellations |
| 10 | RevenueEngine | Subscription Timeline | Revenue Records | Generate recurring revenue and billing events |
| 11 | DirtyDataEngine | Clean Dataset | Messy Dataset | Inject configurable data quality issues |
| 12 | ValidationEngine | Messy Dataset | QA Report | Validate structural, business, and statistical consistency |
| 13 | ExportEngine | Validated Dataset + QA Report | CSV / JSON / Metadata | Export final simulation outputs |

## Engine Contracts

Every engine must satisfy the following rules:

- Accept only the declared input datasets.
- Produce only the declared output datasets.
- Never modify upstream outputs directly.
- Never depend on future pipeline stages.
- Fail fast when required inputs are missing.
- Return deterministic results when using the same random seed.
- Record execution metadata for logging and debugging.

---

## Error Handling Contract

If an engine fails:

1. Stop downstream execution immediately.
2. Log the failed engine.
3. Record the exception details.
4. Preserve intermediate outputs for debugging (optional).
5. Do not export incomplete datasets.

This ensures that only fully validated simulations are written to disk.

# 10. Failure Handling

If a stage fails:

- Stop execution
- Log failure
- Generate diagnostic report
- Prevent export

Partial datasets must never be exported.
